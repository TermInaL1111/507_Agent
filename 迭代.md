# 基于微服务、RAG与智能体（Agent）的高校学生智能服务助手系统

## 1 项目简介

建设一个基于微服务架构的高校学生智能服务助手系统，集成RAG检索增强生成与Agent智能体能力，帮助在校学生快速获取校园信息、办理日常事务，提供智能化的问答与任务执行服务。

### 1.1 目标用户

- **高校学生群体**：使用PC网页端访问系统，希望获得智能问答、事务办理辅助、个性化信息查询等服务
- **后台管理员**：使用电脑熟练，负责新闻资讯管理、用户管理、系统配置与主题风格变更
- **学校主管部门**：关注系统性能、并发承载能力与数据安全

### 1.2 产品功能

#### 1）智能对话功能
- a) 支持自然语言交互，学生可通过对话框输入问题
- b) 基于RAG技术实现校园知识库精准问答（规章制度、培养方案、课程信息等）
- c) Agent智能体支持复杂任务执行：自动生成请假条、解析个人课表、智能办事指南
- d) 支持多轮对话与对话记忆功能

#### 2）用户管理功能
- a) 用户注册、登录、注销功能
- b) JWT安全认证机制
- c) 用户数据隔离与权限控制

#### 3）管理后台功能
- a) 管理员可增删改查新闻资讯
- b) 支持不同时期变更系统主题风格
- c) 用户管理与系统配置

#### 4）知识库管理功能
- a) 支持校园文档上传与向量化存储
- b) 支持知识库内容的动态更新

### 1.3 项目周期

30天内上线（4周交付）。

## 2 用户角色

### 高校学生群体
使用手机熟练，希望该系统能有良好的用户体验，符合智能助手的使用习惯，同时希望能通过Agent智能体自动完成一些繁琐的校园事务办理。

### 后台管理员
使用电脑熟练。系统可扩展性比较强，最好是能引入主题风格，这样可以在不同时期展示不一样的主题页面（功能不变）。

### 学校主管部门
关注系统承载一定的并发量，确保高峰期使用稳定。

## 3 用户故事

为了产生最初的故事列表，团队决定召开一次故事编写工作坊，大家将花费一两个小时，尽可能多写故事。在本案例中，团队讨论后决定以每个角色和用户画像进行故事的编写。

### 高校学生群体

**故事卡1：用户登录注销功能**

**作为** 高校学生  
**我希望** 能够利用账号密码登录和注销系统  
**以便** 安全地访问我的个人信息和收藏内容

**验收条件：**
1. 用户能够使用账号密码成功登录系统
2. 登录成功后显示用户头像和昵称
3. 用户可以点击按钮安全注销
4. 未登录状态下无法访问个人收藏等敏感功能

**故事卡2：对话界面UI设计**

**作为** 高校学生  
**我希望** 首页对话界面UI设计足够友好，支持自然语言输入  
**以便** 能够便捷地与智能助手进行交互

**验收条件：**
1. 首页包含清晰的对话框输入区域
2. 支持自然语言输入，界面响应及时
3. UI设计美观，符合用户使用习惯
4. 在不同分辨率下显示正常

**故事卡3：RAG知识库问答功能**

**作为** 高校学生  
**我希望** 系统支持基于RAG的知识库问答  
**以便** 能够快速准确地获取校园规章制度、培养方案相关信息

**验收条件：**
1. 系统能够准确回答校园规章制度相关问题
2. 系统能够准确回答专业培养方案相关问题
3. 答案引用正确的知识库来源
4. 回答准确率达到可接受水平

**故事卡4：请假条自动生成功能**

**作为** 高校学生  
**我希望** Agent智能体能够理解我的需求并自动生成请假条  
**以便** 快速完成请假条的填写和导出

**验收条件：**
1. Agent能够理解用户生成请假条的意图
2. 根据提示填写请假原因、时间后能够生成规范请假条
3. 生成的请假条格式规范，可导出或复制
4. 支持常见的请假类型（病假、事假等）

**故事卡5：课表解析功能**

**作为** 高校学生  
**我希望** Agent智能体能够解析我上传的个人课表文件  
**以便** 快速提取和查看课程信息

**验收条件：**
1. 支持上传常见的课表文件格式（PDF、图片等）
2. Agent能够正确解析文件内容
3. 准确识别课程名称、时间、地点、教师信息
4. 提取的课程信息展示清晰

**故事卡6：课程信息查询功能**

**作为** 高校学生  
**我希望** 能够查询课程信息  
**以便** 了解课程的时间、地点、教师等详细信息

**验收条件：**
1. 能够通过课程名称查询课程信息
2. 查询结果包含课程时间、地点、教师等信息
3. 查询响应速度快
4. 支持模糊查询和精确查询

**故事卡7：对话记忆功能**

**作为** 高校学生  
**我希望** 对话支持上下文记忆  
**以便** 能够进行多轮连续对话，无需重复说明背景信息

**验收条件：**
1. 系统能够记住对话上下文
2. 支持多轮连续对话，理解指代关系
3. 对话历史记录可查看
4. 上下文记忆在一定时间范围内有效

**故事卡8：智能办事指南功能**

**作为** 高校学生  
**我希望** 系统提供智能办事指南功能  
**以便** 根据我的问题推荐相应的办理流程

**验收条件：**
1. 能够识别用户办事相关的询问
2. 根据问题推荐正确的办理流程
3. 提供清晰的办理步骤说明
4. 涵盖常见的校园办事场景

## 5 发布计划

为了创建发布计划，需要进行以下步骤：
1. 选择一次迭代周期长度
2. 估算速率
3. 对故事进行优先级排序
4. 把故事分配到一个或者多次迭代中

因为新系统的特性是在4周内交付，所以团队决定使用2周长度的迭代。这样团队就能有机会在最后交付期限之前进行两次迭代。把最高优先级的特性放在第一次迭代中，以确保这些特性一定能完成。在第一次迭代之后，团队就能评估他们的速率，并决定可以将多少工作放在第二次迭代。

## 6 实现

在软件开发过程中，可采用TDD（测试驱动开发）的方法，开发不断满足测试人员编写的单元测试与集成测试来完成某个故事和Issue。并在TAPD平台跟踪故事或者Issue完成情况。

### 技术架构

- **前端**：PC网页端，使用现代前端框架
- **后端服务**：
  - FastAPI：提供核心智能服务，集成LangChain RAG与Agent智能体
  - Django：独立实现用户管理、权限控制与数据持久化
- **数据库**：向量数据库支持RAG检索，关系型数据库支持业务数据
- **AI能力**：RAG检索增强生成 + Agent智能体任务规划与工具调用

## 9 项目总结

### 一句话总结
本系统融合 **RAG 知识库问答 + Agent 智能体任务执行 + 微服务架构**，面向高校学生提供智能答疑、自动生成请假条、课表解析与课程服务等一体化功能，是一个完整覆盖软件工程实践要求的 AI 应用项目。

### 关键技术
- ✅ **微服务架构**（Django + FastAPI）
- ✅ **RAG 检索增强生成**
- ✅ **Agent 智能体**（工具调用、任务执行）
- ✅ **前后端分离（PC 网页端）**
- ✅ **特色功能：请假条生成、课表导入**
- ✅ **JWT 认证、对话记忆、持久化**
- ✅ **基于开源项目二次开发**
